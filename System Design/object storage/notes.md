### Key Concepts and Definitions

| Term                   | Definition / Explanation                                                                                      |
|------------------------|--------------------------------------------------------------------------------------------------------------|
| **Object Storage**     | A storage architecture optimized for large files (binary large objects or blobs), such as videos, photos, JSON files, and large text files. |
| **Blob (Binary Large Object)** | Large files typically ranging from megabytes to gigabytes, stored as a collection of bytes.                |
| **Relational Database** | Databases like PostgreSQL designed for small, frequently changing records with complex queries, joins, and transactions. Not suited for large static files. |
| **Immutable Writes**   | Object storage does not modify files in place but creates new versions or overwrites whole files, simplifying concurrency control.|
| **Flat Namespace**     | Object storage organizes files by unique keys (strings) rather than hierarchical folder structures, allowing faster lookups.|
| **Redundancy**         | Files are replicated or erasure-coded across multiple servers, racks, or even data centers, ensuring extremely high durability (commonly 11 nines, or 99.999999999%).|
| **Pre-signed URL**     | Temporary URLs generated by the storage system allowing clients to upload/download files directly without routing through the application server.|
| **Multi-part Upload**  | A technique where large files are split into smaller chunks (e.g., 5 MB), uploaded individually, and then reassembled by the storage system. |

### Why Not Store Large Files in Traditional Databases?

- Traditional relational or OLTP databases are optimized for **small, frequently updated data** requiring rich queries.
- Large files (e.g., 4 MB images) span hundreds of internal 8 KB pages, causing:
  - **Performance degradation**: Queries fetching small user data get slowed by scanning large blobs.
  - **Increased memory pressure** and unnecessary data loading.
  - **Replication overhead**: Each write replicates large blobs across all database replicas, increasing latency and bandwidth usage.
  - **Backup and restore inefficiency**: Backups become bloated with large static files, drastically increasing restoration times and operational risk during failures.
  
**Conclusion:** Storing blobs in traditional databases results in poor performance and high operational cost. Object storage is designed specifically to handle these challenges.

### How Object Storage Works (High-Level Overview)

| Component          | Role                                                                                               |
|--------------------|--------------------------------------------------------------------------------------------------|
| **Storage Nodes**  | Physical servers where files (objects) are stored on disks, typically in large data centers.       |
| **Metadata Service** | Maintains an index mapping object keys to storage locations (servers). Clients query this service to find where a file resides. |
| **Client**          | Initiates requests to fetch or upload files by first querying metadata and then streaming data directly to/from storage nodes. |

- Files are stored across multiple servers for **redundancy**.
- Clients receive file location information from metadata service, then directly stream data from the appropriate storage node.
- This architecture enables **cheap, scalable, and durable storage**.

---

### Important Features of Object Storage

- **Flat Namespace**: Unlike hierarchical file systems with folders, object storage uses a flat key-value system where filenames are unique strings. UI tools may show folders, but these are abstractions.
- **Immutable Writes**: Files cannot be partially updated; updates involve creating new versions or overwriting entire objects, simplifying concurrency and locking.
- **Redundancy and Durability**: Objects are replicated or erasure-coded across multiple nodes and often across data centers to achieve extremely high durability (e.g., 99.999999999%). This allows automatic recovery from node failures without data loss.

### Recommended Design Patterns for Using Object Storage

- **Store large files (blobs) in object storage only**.
- **Store metadata about these files (e.g., URLs, file details) in a traditional relational database** for fast querying and rich functionality.
  
Example use case:
- A social media platform stores user posts and metadata such as post ID, creator, and text in a relational database.
- The post's associated photo or video is stored separately in object storage.
- The database contains a URL linking to the object storage file.
- Clients first get metadata from the database, then fetch the actual file directly from object storage.

- Application logs and output files.

### Uploading and Downloading Files with Object Storage

- **Pre-signed URLs** allow clients to upload or download files directly to/from object storage without routing through the application server.
  - This avoids server bandwidth bottlenecks and scaling issues.
  - The server generates a URL with limited-time permissions, which the client uses to perform HTTP PUT/GET operations directly.
- This pattern is widely used in industry and often discussed in system design interviews.

---

### Handling Large Files: Multi-part Upload

- Uploading very large files is challenging due to HTTP size limits imposed by browsers, servers, and gateways.
- Object storage solutions (e.g., Amazon S3) support **multi-part uploads**, where:
  - Large files are broken down into smaller chunks (e.g., ~5 MB per chunk).
  - Chunks are uploaded individually, possibly in parallel.
  - The object storage service reassembles the chunks into the final complete file.
- This method improves reliability and efficiency of large file uploads.
